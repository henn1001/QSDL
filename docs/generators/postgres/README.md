# PostgreSQL generator

The PostgreSQL generator produces a SQL migration script that creates tables, columns, and foreign key constraints from your QSDL schema.

## Output

The generator creates a single SQL file (default: `V1_0_0__baseline.sql`) containing:

- **Type tables**: One table per `type` (domain object) in your schema (with primary key, unique uid, version counter).
- **Base types**: Inlined (flattened) into type tables by default, or stored as JSONB columns if marked with `@opaque`.
- **Composition relationships**: Foreign keys that enforce parent-child hierarchy constraints.
- **Aggregation relationships**: Auto-generated join tables for many-to-many relationships.
- **Enum constraints**: Check constraints for enum fields.
- **Column types**: Scalars are mapped to PostgreSQL types (TEXT, BIGINT, BOOLEAN, etc.).

## Configuration

| Name           | Type   | Default                  | Description                                                         | Required |
| -------------- | ------ | ------------------------ | ------------------------------------------------------------------- | -------- |
| `file_name`    | string | `"V1_0_0__baseline.sql"` | Output SQL filename (following Flyway migration naming convention). | No       |
| `table_prefix` | string | `"T_"`                   | Prefix prepended to all table names (converted to uppercase).       | No       |

## Scalar-to-PostgreSQL Type Mapping

| QSDL Scalar | PostgreSQL Type            |
| ----------- | -------------------------- |
| `Int`       | INTEGER                    |
| `Long`      | BIGINT                     |
| `Float`     | REAL                       |
| `Double`    | DOUBLE PRECISION           |
| `String`    | TEXT                       |
| `Boolean`   | BOOLEAN                    |
| `Date`      | DATE                       |
| `Datetime`  | TIMESTAMP                  |
| `Object`    | JSONB                      |
| `Enum`      | TEXT with check constraint |

## Relationship Handling

### Composition (`@composition`)

Composition fields create a foreign key constraint where the parent "owns" the children. The field must be a required array (`[TypeName]!`).

**Example schema:**

```qsdl
type User {
  name: String!
  contacts: [Contact]! @composition
}

type Contact {
  email: String!
}
```

**Generated SQL:**

```sql
CREATE TABLE IF NOT EXISTS T_USER (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid VARCHAR UNIQUE,
  iv INTEGER,
  name TEXT NOT NULL,
  contacts_id BIGINT NOT NULL,
  FOREIGN KEY (contacts_id) REFERENCES T_CONTACT(id)
);

CREATE TABLE IF NOT EXISTS T_CONTACT (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid VARCHAR UNIQUE,
  iv INTEGER,
  email TEXT NOT NULL
);
```

### Aggregation (`@aggregation`)

Aggregation fields create join tables for many-to-many relationships. The field must be a required array (`[TypeName]!`).

**Example schema:**

```qsdl
type Student {
  name: String!
  courses: [Course]! @aggregation
}

type Course {
  title: String!
}
```

**Generated SQL:**

```sql
CREATE TABLE IF NOT EXISTS T_STUDENT (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid VARCHAR UNIQUE,
  iv INTEGER,
  name TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS T_COURSE (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid VARCHAR UNIQUE,
  iv INTEGER,
  title TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS T_STUDENT_COURSES (
  student_id BIGINT NOT NULL,
  course_id BIGINT NOT NULL,
  PRIMARY KEY (student_id, course_id),
  FOREIGN KEY (student_id) REFERENCES T_STUDENT(id),
  FOREIGN KEY (course_id) REFERENCES T_COURSE(id)
);
```

## Base Type Handling

Base types can be stored in two ways: **flattened** (default) or as **JSONB** using the `@opaque` directive.

### Default: Flattening

By default, base type fields are **flattened** into the type table. Fields from the base are inlined with prefixed column names.

**Example schema:**

```qsdl
base Address {
  street: String!
  city: String!
}

type User {
  name: String!
  homeAddress: Address
}
```

**Generated SQL:**

```sql
CREATE TABLE IF NOT EXISTS T_USER (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid VARCHAR UNIQUE,
  iv INTEGER,
  name TEXT NOT NULL,
  home_address_street TEXT NOT NULL,
  home_address_city TEXT NOT NULL
);
```

Each base field is prefixed with the parent field name (e.g., `homeAddress.street` → `home_address_street`).

### Opaque: JSONB Storage

Mark a base field with `@opaque` to store it as JSONB, bypassing flattening. Useful for flexible or nested structures.

**Example schema:**

```qsdl
base Address {
  street: String!
  city: String!
}

type User {
  name: String!
  primaryAddress: Address @opaque
  secondaryAddress: Address @opaque
}
```

**Generated SQL:**

```sql
CREATE TABLE IF NOT EXISTS T_USER (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid VARCHAR UNIQUE,
  iv INTEGER,
  name TEXT NOT NULL,
  primary_address JSONB,
  secondary_address JSONB
);
```

Each opaque base field becomes a single JSONB column storing the entire base structure.

### Nested Base Flattening

Flattening works recursively. If a base extends another base, all fields are flattened with cumulative prefixes.

**Example schema:**

```qsdl
base Location {
  latitude: Float!
  longitude: Float!
}

base Address extends Location {
  street: String!
  city: String!
}

type Warehouse {
  name: String!
  location: Address
}
```

**Generated SQL:**

```sql
CREATE TABLE IF NOT EXISTS T_WAREHOUSE (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid VARCHAR UNIQUE,
  iv INTEGER,
  name TEXT NOT NULL,
  location_latitude REAL NOT NULL,
  location_longitude REAL NOT NULL,
  location_street TEXT NOT NULL,
  location_city TEXT NOT NULL
);
```

## Generated Table Structure

Every table includes three system columns:

- `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`: Surrogate key.
- `uid VARCHAR UNIQUE`: Unique identifier (typically UUID in application).
- `iv INTEGER`: Version counter (for optimistic locking).

User-defined fields follow these system columns.

## Naming Conventions

- **Table names**: Converted to UPPER*SNAKE_CASE and prefixed with `table_prefix` (default `T*`).
  - Example: `User` → `T_USER`, `CustomerOrder` → `T_CUSTOMER_ORDER`
- **Column names**: Converted to lower_snake_case.
  - Example: `firstName` → `first_name`, `userId` → `user_id`
- **Join tables**: Named as `T_<PARENT>_<FIELD>` for aggregation relationships.
  - Example: `Student.courses` → `T_STUDENT_COURSES`

## Flyway Compatibility

The default filename follows the Flyway migration naming convention (`V<version>__<description>.sql`), making the output directly compatible with Flyway-based deployment pipelines.
